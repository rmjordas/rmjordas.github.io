Hello and welcome to my website.

I’ve been working on some projects that use Gatsby and I wanted to try using it in my own website. Well, here are my notes and observations.

Not much change from before, in terms of the colors and typography. Just some minor tweaks here and there. Although, there are big changes to the layout and content: I added a way to pull data from external sources, rather than hard-coding those in my repository.

This is quite a big deal because adding or removing content using the old way will require a lot of steps and it’s getting tedious.

In this article, I’ll try to describe what tools I used for this project, my experience with said tools, and some discussion on what features I’d like to add.

My Takeaways

Gatsby

A big part of the reason why I choose to learn Gatsby instead of some other library is mostly because it popular. You get a lot of resources (guides, tutorials, and StackOverflow posts, duh!) when you choose to use a popular tool.

Is it the right choice, though? I’m not yet sure.

My overall impression of Gatsby is still mixed and it’s probably because I’ve yet to explore even a quarter of what it’s capable of. Of that small nugget that is my knowledge of Gatsby, I can form a preview of what I like and don’t like about it.

On the one hand, I like Gatsby because it helps me avoid doing things that are easy to get wrong.

The developer experience is superb—it has good defaults when building a static website, it comes with hot module replacement out of the box, Webpack is already useful without tweaking a config somewhere, adding support for popular tools is just a plugin away, and, to top all that, the documentation is a great read.

You’ll get smaller bundles, progressively enhanced images, and generally fast loading website. All with little to no effort, especially if you use a starter repository.

But, on the other hand, it’s just too magical and very tedious to configure.

First, what do I mean by “magical”?

I don’t like that everything is abstracted away from me. It reminds me a lot, when I was using jQuery when I was just a wee coder lad. I didn’t like jQuery then because I felt like it was hindering my ability to learn JavaScript. Maybe this aversion to magical tools is just a remnant of that experience.

“But wait”, you say, “everything is open source. Just take a day or two to read the source!”

“Yeah, I should do that. But not now.”

The Gatsby project has been adding a lot of new stuff. There’s Gatsby Cloud, Theme UI, and who know what other lock-in feature they’ll add next. And, more importantly, my focus, right now, is learning web and software patterns; and core technologies, like the “holy trinity” (HTML, CSS and JavaScript), and TypeScript. I’m also currently building something that focuses on the back-end. But I digress.

Using pre-configured starter repository is not bad practice in any way. In fact, you can go from zero to deploy, quickly, if that’s your goal. But when you now need to reach for a complex feature, things get a little hairy.

The number one thing that irks me when configuring Gatsby, is the lack of native TypeScript support. It’s not possible to write Gatsby configuration files in TypeScript without some hacky workaround.

I have a section on TypeScript later that will talk more about this. Also, they’re working on it now because this issue (gatsbyjs#18983) is still open and has some recent activity.

I don’t mean to sound like Gatsby is poor software because it lacks a certain tool that I want to use. It is wonderful, for the most part, and I would still keep using it as long it doesn’t constantly push breaking features.

TypeScript

One thing I tried to do before starting the re-write is to determine if I can work on this project without TypeScript.

It’s quite resonable to use just JavaScript for small projects like this. Using the latest JavaScript specification is actually better compared to when I was first starting to learn.

But I realized, it’s not going to work for me.

Because components will probably have to depend on build-time queries (using GraphQL) for it to have data show, I have to accept that I won’t get Intellisense for the return value.

Consider this query:

const data = useStaticQuery(graphql`
  query {
    allRepos {
      nodes {
        id
        stargazers {
          totalCount
        }
      }
    }
  }
`);

It’s a pretty simple example. In fact, you might argue that adding types here is not needed. It’s simple enough to remember that the id is probably a string, and stargazers is some object that has a totalCount property which is number.

Pass this return value to child components and this is still manageable.

const Child = ({ id, stargazers }) => {
  <li id={id}>
    {stargazers.totalCount}
  </li>
}

const Parent = () => {
  const data = useStaticQuery(/* query */);

  return (
    <ul>
      {data.allRepos.nodes.map(({ id, ...childProps }) => (
        <Child {...childProps} key={id} />
      ))}
    <ul>
  )
};

But when you get a query with a lot of properties, and your drilling these properties down one or two levels, you’ll see how much you’ll regret not adding types to your components.

And add to the fact that I may want to move around some components, it’s hard to keep track of what a component needs, in terms of props.

So, rather than fighting my tools by not giving it enough to help me, I decided it’s probably better to use TypeScript in this project.

I’m liking the outcome so far.
